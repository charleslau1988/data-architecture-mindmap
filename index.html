▶ 一、MSSQL：用对了是利器，用错了是坑4个要点
▶ 1.1 索引：别盲目加，先看访问模式
聚簇索引一个表只能有一个，选哪个字段取决于业务怎么查性别字段建索引纯属浪费——区分度太低隐式类型转换是隐形杀手：字段varchar传参int，索引直接废掉WHERE YEAR(CreateDate)=2020 函数包列索引白建，改用范围查询统计信息过期会导致优化器选错执行计划
性能调优心得：先看 Logical Reads，SQL Server 性能瓶颈本质是 IO。Logical Reads 从 10 万降到 1 万，性能提升 10 倍以上。执行计划从右往左读，找代价最大的节点。
-- 调优必备三件套
SET STATISTICS IO ON;
SET STATISTICS TIME ON;
-- 然后看执行计划，重点找 Index Scan / 全表扫描
▶ 1.2 存储过程 vs 函数：各司其职
业务逻辑封装用存储过程，计算逻辑封装用函数ETL批量清洗和写入用存储过程——可以控制事务、处理异常SELECT里做字段级转换用函数——可直接嵌在查询里
-- 函数：字段级计算，SELECT里直接用
CREATE FUNCTION fn_OrderTotal(@OrderID INT)
RETURNS DECIMAL(18,2) AS
BEGIN
    RETURN (SELECT SUM(Amount) FROM OrderDetail WHERE OrderID = @OrderID)
END

-- 存储过程：批量操作，带事务控制
CREATE PROCEDURE sp_ProcessOrders @Status INT AS
BEGIN
    BEGIN TRANSACTION;
    UPDATE Orders SET Status = @Status WHERE ...
    COMMIT;
END
▶ 1.3 事务与锁：Serializable 别轻易用
MSSQL默认Read Committed，绝大多数场景够用高并发写入时建议开启RCSI（读已提交快照隔离），减少锁竞争死锁不是故障，是并发控制的正常表现核心应对：统一访问顺序 + 缩短事务时间
-- 开启RCSI
ALTER DATABASE DBName SET READ_COMMITTED_SNAPSHOT ON;

-- 乐观并发（版本号控制）
UPDATE Products
SET Stock = Stock - 1, Version = Version + 1
WHERE ID = @ID AND Version = @ExpectedVersion;

IF @@ROWCOUNT = 0
    RAISERROR('数据已被修改，请重试', 16, 1);
▶ 1.4 分区表：5000 万行以上再考虑
单表超5000万行、需要历史归档、需要并行操作——满足两个就该分区按时间分区最常见，按月或按季度查询一定要带分区字段，否则扫所有分区比不分区还慢
CREATE PARTITION FUNCTION pf_OrderDate (DATETIME)
AS RANGE RIGHT FOR VALUES ('2020-01-01', '2021-01-01', '2022-01-01');

CREATE PARTITION SCHEME ps_OrderDate
AS PARTITION pf_OrderDate
TO (FG2020, FG2021, FG2022, FGArchive);
▶ 二、ETL 管道：先落地再清洗，别在采集层做逻辑7个主题
▶ 2.1 SSIS 的架构理解
控制流管"做什么"（执行顺序、条件分支）数据流管"怎么做"（抽取、转换、加载）数据清洗逻辑放数据流里，控制流只做调度Source: OLE DB Source, Flat File Source, Excel SourceTransformation: Derived Column, Lookup, Conditional Split, Merge Join, AggregateDestination: OLE DB Destination, Flat File Destination
▶ 2.2 多平台数据整合的分层架构
分层的核心价值：出问题能定位到具体层，不用全局排查
层级存放内容数据特征谁会用
Source各平台原始数据字段乱、格式杂、未清洗极少
Stage/ODS采集原始表+异常表零清洗、加溯源字段数据工程师
DW统一维表+统一事实表已标准化、已去重分析师+工程师
Data Mart业务主题宽表已反规范化、业务友好业务方+分析师
数据流向永远单向：Source → Stage → DW → Mart，不允许反向回写Stage层不建维度表：只有原始表和异常表DW层不加业务字段：纯整合，出问题能溯源到StageMart层是交付层：按业务主题构建，业务方直接查询
▶ 2.3 RPA 数据：质量差是常态，不是意外
常见问题：字段缺失、格式不一致、乱码、采集遗漏、重复数据、字段错位先探底：采集100条样本看数据质量基线分两阶段清洗：第一阶段标准化格式，第二阶段业务逻辑清洗加质量监控：采集成功率、数据量波动、异常率都要盯
最重要的一条：不要在采集层做太多逻辑。RPA脚本维护成本很高，采集层只负责"拿数据"，清洗留给ETL。
▶ 2.4 脏数据：不丢弃，但必须隔离
核心原则：不让异常阻断ETL，但也不能让异常数据污染目标表。
正常数据 → 目标表 | 脏数据 → 异常表（打标原因）→ 告警通知
类型处理方式
字段缺失默认值填充或标记NULL，可以进目标表但可查
格式错误尝试标准化，失败则入异常表
主键冲突判断是重复还是真正冲突，分别处理
业务逻辑错误入异常表，业务方确认后再处理
完全无法解析入异常表，保留原始数据以便回溯
有些"脏数据"其实是业务边界情况，直接丢弃会漏数据处理异常表时必须让业务方参与
▶ 2.5 增量 vs 全量：优先时间戳方案
场景选全量选增量
数据量<10万行✅—
数据量大、变化比例小—✅
源表无时间戳/自增ID✅—
初次建仓✅—
增量实现优先级：时间戳字段 > 自增ID > CDC > 触发器 > 日志解析时间戳方案侵入性最小每次跑完增量一定要核对：增量行数+已累计行数=总行数
DECLARE @LastLoadTime DATETIME = (SELECT MAX(LoadTime) FROM TargetTable);
INSERT INTO TargetTable
SELECT * FROM SourceTable WHERE UpdatedAt > @LastLoadTime;
▶ 2.6 调度与监控：告警必须分级
INFO：任务开始/完成（记录但不打扰）WARN：数据量波动±20%、采集成功率低（关注但不需要立即处理）ERROR：任务失败、异常率超标（必须立即响应）
告警不分级 = 告警疲劳 = 所有人免疫 = 等于没有告警
-- 任务日志表设计
INSERT INTO ETL_Log (PackageName, StartTime, EndTime, Status, RowsProcessed, ErrorMessage)
VALUES ('pkg_OrderETL', @Start, GETDATE(), 'Success', @RowCount, NULL);
▶ 2.7 数据一致性：先对账后入库
入仓前：ETL前查源表总数，与采集后数量对比，差异超阈值则暂停ETL中：用事务保证同批次原子性，每步记日志ETL后：核对源表行数 = ETL成功行数 + 异常行数日常巡检：每日自动跑一致性校验，输出报告
核心理念：数据进入正式表之前先跑一致性校验，账对不平就告警，不入库。下游拿到的数据必须是经过质量认证的。
▶ 三、数据建模：选型看场景，别一刀切5个主题
▶ 3.1 星型 vs 雪花：90%的场景选星型
星型：1事实+N维度，维度表不进一步拆分，JOIN少查询快维护简单雪花：维度表再规范化成多层，存储省但JOIN多查询慢绝大多数分析场景星型就够了除非维度表有200+字段且有明确层级，否则优先星型
▶ 3.2 宽表：空间换时间，用在最终交付层
把多个维度字段合并到一张大表，减少JOIN，提升查询性能宽表的正确位置：分析师的最终查询表，不是ETL的中间层最好用视图实现而非物理表，保证数据实时性数据量大时用物化视图或定时刷新用户级别变了，宽表怎么更新？历史数据怎么处理？设计阶段要想清楚
▶ 3.3 关系建模：1:1 / 1:N / M:N 各有套路
1:1：直接合并或加外键1:N：多方加外键（订单→订单明细）M:N：必须用关联表（Bridge Table）
M:N关键细节：关联关系本身如果有业务属性（购买数量、价格），关联表必须有。如果只是ID映射，可以简化。
▶ 3.4 3NF vs 反规范化：分层处理
ETL中间层保持3NF，BI查询层反规范化/宽表中间层如果反规范化，出问题时很难定位根因最终交付层宽表化，分析师用起来快ETL层反规范化是给自己挖坑
▶ 3.5 事实表 vs 维度表：看业务语义
描述"发生了什么"→ 事实表描述"是什么"→ 维度表超过30个字段的基本是维度表，不到10个的基本是事实表维度退化：订单号、发票号等"只做追溯用"的标识符直接放事实表需要分析的（如订单状态），才单独建维度表

SN是产品的物理唯一标识，不随渠道变化同一个产品在不同地方叫法不同，但SN不变新渠道接入只需加映射关系，不用改事实表结构设计核心：SN Hub ↔ ID Mapping ↔ 统一事实表

▶ 五、跨平台 ID 映射：最难的不是建表，是保证映射质量6个主题
▶ 5.1 Hub-Spoke 映射模型

▶ 5.2 SN/PN/userid/openid 的关联层次
第一层：产品关联（SN ↔ PN）
PN是规格型号，SN是生产批次，一个PN对应多个SN

第二层：用户关联（userid ↔ openid）
淘宝openid真正标识是unionid，一个userid对应1~N个openid

第三层：购买关系（SN + userid）
通过订单/售后记录关联，M:N关系
▶ 5.3 ID 映射的质量保障
强标识符优先：SN、手机号做精准匹配模糊匹配加置信度：同一IP/地址关联，置信度低于0.8需人工审核映射历史可追溯：SCD Type 2记录变更历史，支持回溯无法关联的数据进异常表：不能强行匹配导致数据错误持续迭代：覆盖率从60%提升到95%是长期过程
▶ 5.4 常见 ID 问题的处理
问题原因解决方案
SN重复扫码录入错误或产品窜码加校验规则（校验位）
openid关联不上用户换公众号导致openid变了引导用户登录获取unionid
订单无SN散装产品按件卖，无SN用订单号+商品编号替代
userid丢失RPA采集时未登录态采集端补录或关联收货人信息
▶ 5.5 去重：找到业务键是核心
去重不是技术问题，是业务问题——什么情况下两条记录算"重复"？通常是多字段组合（SN+OrderCode+Amount）推荐方案：Hash去重
-- 计算业务键Hash
SELECT *, HASHBYTES('SHA2_256',
    CAST(ProductSN AS NVARCHAR(MAX)) + CAST(OrderDate AS NVARCHAR(MAX))
) AS BusinessKeyHash
FROM Orders;

-- 去重：只保留最早录入的
WITH Ranked AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY BusinessKeyHash ORDER BY CreateTime ASC
    ) AS rn
    FROM Orders
)
DELETE FROM Ranked WHERE rn > 1;
▶ 5.6 主键唯一性的保障与踩坑
保障手段：数据库约束 > 应用层逻辑——永远在数据库层面加约束，比应用层更可靠。
-- ETL幂等性用MERGE保证
MERGE INTO dim_Products AS target
USING staging_Products AS source
ON target.ProductSN = source.ProductSN
WHEN MATCHED THEN UPDATE SET ProductName = source.ProductName
WHEN NOT MATCHED THEN INSERT (ProductSN, ProductName, ...) VALUES (...);
坑1：源数据本身有重复——ETL前加去重步骤坑2：历史迁移主键冲突——各平台加前缀（JD_/CITIC_）坑3：复合主键中NULL导致去重失效——用ISNULL()转成特殊值
▶ 六、数据质量治理：不是一次性项目，是持续运营机制5个主题
▶ 6.1 四维度监控框架
维度监控内容核心指标告警阈值
完整性数据缺失空值率、主键缺失率空值率>1%
一致性跨系统一致外键引用失败率、枚举值合法率失败率>0.5%
时效性更新及时数据延迟时长延迟>8h
准确性数据正确格式错误率、业务规则违背率错误率>5%严重告警
-- 完整性
SELECT COUNT(*) AS TotalRows,
    SUM(CASE WHEN ProductSN IS NULL THEN 1.0 ELSE 0 END) / COUNT(*) AS NullSNRate
FROM fact_Orders;

-- 时效性
SELECT MAX(OrderDate) AS LatestOrderDate,
    DATEDIFF(HOUR, MAX(OrderDate), GETDATE()) AS DataDelayHours
FROM fact_Orders;

-- 准确性
SELECT SUM(CASE WHEN Amount < 0 THEN 1 ELSE 0 END) AS NegativeAmountCount,
    SUM(CASE WHEN OrderDate > GETDATE() THEN 1 ELSE 0 END) AS FutureDateCount
FROM fact_Orders;
▶ 6.2 三层防护体系
第一层：ETL过程中实时打标——异常数据标记，不阻断流程第二层：ETL完成后跑质量校验任务——核对四维度指标第三层：每周/月出质量趋势报告——关注趋势，不只是当前值
P0（必须立即处理）P1（当天处理）P2（下周处理）
▶ 6.3 异常处理 SOP
闭环：识别（Detect）→ 隔离（Isolate）→ 分析（Analyze）→ 处置（Resolve）
类型时效
字段缺失（可填充）当天
格式错误（可转换）当天
主键冲突当天
业务逻辑错误24h（需业务方确认）
数据丢失48h（补采或手工补录）
系统性错误立即（修复ETL后重跑）
关键：每次异常都要记录根因和处置方式，每月出异常分析报告。同一问题重复出现=ETL流程有漏洞，需要系统性修复，而不是每次靠人工。

▶ 七、无 API 场景的 ETL 架构设计3个主题
▶ 7.1 无API场景架构设计
架构层次：

数据采集层（RPA）
  └─ 模拟人工操作获取页面数据
  └─ 难点：结构不固定、字段缺失、需登录态

数据缓存层（Raw）
  └─ RPA输出直接落原始表，零清洗
  └─ 加采集时间戳、批次号，方便溯源

数据清洗层（Staging）
  └─ 字段映射标准化，脏数据打标入异常表

数据关联层（ODS/DW）
  └─ 跨平台ID映射，写入统一事实表

监控告警
  └─ 采集成功率、数据量波动、异常字段
核心心法：没有API的场景，先拿数据再清洗。采集层别做逻辑，否则RPA脚本维护成本爆炸。
▶ 7.2 新数据源接入的标准化流程
需求分析 → 数据探底 → 主键映射设计 → ETL开发 → 质量校验 → 上线交付
不是每次都从零设计，先输出标准模板，让后续接入效率提升交付时必须带：字段映射表、数据字典、更新日志，声明SLA
▶ 7.3 多平台数据不一致的排查路径
从目标往源排查：

1️⃣ 查目标表写入逻辑：先到先写vs后到覆盖？去重规则误删了有效数据？
2️⃣ 查ID关联环节：主键映射表完整吗？ID冲突导致记录丢失？
3️⃣ 查ETL清洗过程：字段映射搞混了？清洗规则不一致（大小写、空格、编码）？
4️⃣ 查源头数据：各平台原始数据本身就不一致？RPA采集丢了字段？
定位根因后，先修数据，再修流程，加监控防止再发
▶ 八、与业务方沟通：说人话翻译对照
▶ 8.1 技术术语 vs 业务语言
技术术语业务语言
事实表交易记录表
维度表产品信息表
外键关联通过产品编号，把各平台数据串起来
数据去重同一个人多个订单，去掉重复只算一次
ETL管道从各个平台自动搬数据进来的流程
数据质量监控每天自动检查数据有没有问题
话术模板：
"你的需求是看每个产品的全渠道销量。我设计的方式是以产品编号为核心，把JD、淘宝、花旗的数据都关联到同一张表里。你看到的每一行就是一个产品在一个渠道的完整数据，不需要手工匹配，分析师直接出报表。"
▶ 九、关键经验清单正确决策+踩坑+认知
▶ ✅ 做过的正确决策
分层架构（Raw→Staging→DW）：出问题能定位到具体层，不用全局排查SN做核心Hub：新渠道接入只加映射字段，不动事实表异常数据隔离不入目标表：避免脏数据污染下游ETL幂等性用MERGE：重复执行不会出问题告警分级：避免告警疲劳数据质量日报+Traffic Light：业务方看得懂，才会信任数据
▶ ❌ 踩过的坑
隐式类型转换导致索引失效：SQL不报错但索引废了RPA采集层做清洗逻辑：页面改版后脚本全崩，维护成本爆炸脏数据直接丢弃：丢了业务边界数据，分析结果偏差ETL中间层反规范化：出问题排查成本极高复合主键有NULL：唯一约束不生效，重复数据悄悄混入历史数据迁移不做主键冲突检测：上线直接报错告警不分级：一堆告警没人看，等于没有监控
▶ 💡 最重要的认知
数据质量监控是持续运营机制，不是一次性项目数据库约束 > 应用层逻辑，永远在数据库层面保证规则先对账后入库，下游拿到的数据必须是经过质量认证的脏数据不丢弃，但必须隔离RPA数据质量差是常态，不是意外——架构要为这个常态设计统一ID自增生成，不依赖任何平台ID映射覆盖率从60%提升到95%是长期过程，不可能一步到位
